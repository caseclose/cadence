# Cadence 端到端加密（E2EE）说明

> 可以把本文发给担心隐私的用户：说明**云端看不到任务明文**，以及密钥如何工作。

## 一句话

**任务标题、备忘录（Markdown）等内容在离开浏览器之前就已加密；Supabase / 仓库运维只能看到密文，无法还原你的私钥。**

## 威胁模型

| 攻击面 | 是否防护 |
|--------|----------|
| 数据库泄露、Supabase 管理员查看 | ✅ 仅密文 |
| 传输窃听（HTTPS 之上） | ✅ 密文 payload |
| 同一账号在其他设备用正确密码登录 | ✅ 可解密（设计如此） |
| 忘记登录密码 | ❌ **无法恢复**任务明文（无后门） |
| 浏览器被植入恶意脚本 | ❌ 与任何 Web 应用相同 |
| 本地设备被物理控制 | ❌ 内存中可能有解密密钥；解锁后约 6 小时内本机 localStorage 也可能有 DEK 会话 |

## 密钥设计（公钥 / 私钥 + 数据密钥）

注册或首次启用加密时，浏览器会：

1. **生成 RSA 密钥对**（2048 位）
   - **公钥**：写入账号 metadata（`e2ee_public_jwk`），用于包裹数据密钥
   - **私钥**：用 **登录密码 + PBKDF2（12 万次）** 派生的 AES 密钥加密后，存入 metadata（`e2ee_priv_enc`）；**明文私钥不离开本机内存**

2. **生成随机 AES 数据密钥（DEK）**，专门加密任务 JSON
   - DEK 经 **RSA 公钥** 包裹后存 metadata（`e2ee_dek_wrapped`）
   - 只有 **RSA 私钥**（用你的密码解开）才能还原 DEK

3. **每条任务**写入 Supabase 时：
   - `tasks.enc` = AES-GCM 密文（含 IV）
   - 表里的 `title` 等字段仅为占位符 `[e2ee]`，**不含真实内容**

登录时：输入密码 → 解密私钥 → 解开 DEK → 在浏览器内解密任务。

解锁成功后：本机将 DEK 会话缓存在 `localStorage`，约 **6 小时内刷新页面无需再输密码**；退出登录或过期后会清除。DEK 仍不离开本机，云端看不到。

## 多设备

- 密钥材料备份在 **你自己的** Supabase 用户 metadata 中（已是加密形态）
- 新设备：登录 + 输入同一密码 → 自动解密 metadata → 同步任务
- **密码是唯一恢复手段**；我们不托管密码，也没有 master key

## 本地模式（未登录 / 未配 Supabase）

- 数据只在浏览器 `localStorage`，不走云端
- 不启用上述 E2EE 流程（无服务器可防）

## 给运维 / 开发者的说明

- 即使拥有 `service_role` key，也只能读到 `enc` 密文和加密后的私钥备份
- **例外（后台推送）**：为调度 Web Push，云端可读每个任务的 `next_fire_at` 与 `state`；推送文案不含任务内容。见 [PUSH.md](PUSH.md)
- 不要在服务端实现「解密接口」——那会破坏 E2EE 保证
- 算法与实现见 [`src/crypto/e2ee.ts`](../src/crypto/e2ee.ts)、[`src/crypto/keyring.ts`](../src/crypto/keyring.ts)

## 局限

- 密码即密钥：密码太弱可被离线暴力尝试 metadata 中的私钥封装（已用 PBKDF2 减缓）
- 无密钥恢复：丢密码 = 丢云端密文任务（本地缓存若还在，可先导出 JSON 备份）
- 旧账号在**首次用新版本登录**时会自动启用 E2EE，历史明文任务会在下次同步时改为密文上传
